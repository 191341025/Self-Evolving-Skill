# 贝叶斯反馈机制设计：α/β 从哪来

> Self-Evolving Skill 设计文档
> 状态：设计草案，待评审
> 前置依赖：`research/design/decay-model-notes.md`（公式推导）

---

## 1. 问题陈述

衰减模型已从固定 λ 升级为贝叶斯自适应 λ：

```
原公式：C(t) = e^(-λ_base × t)             ← λ 是常量，只靠时间衰减
新公式：C(t) = e^(-λ_base × (β+1)/(α+1) × t)  ← λ 被使用反馈动态调整
```

公式本身已验证，但它依赖一个前提：**agent 必须能在使用知识后产生 α（成功）或 β（失败）的反馈信号**。

本文档要回答三个问题：
1. 什么算"成功"，什么算"失败"？
2. 在 db-investigator 场景中，具体有哪些反馈来源？
3. α/β 存在哪里、谁来写入、什么时候写入？

---

## 2. 反馈信号的定义原则

### 2.1 不是所有使用都能产生反馈

知识被"读取"不等于被"验证"。只有以下情况才构成有效反馈：

| 情况 | 是否构成反馈 | 原因 |
|------|------------|------|
| agent 读了 schema_map 中的 JOIN 路径，执行 SQL 成功 | **是，α+1** | SQL 成功 = JOIN 路径确实正确 |
| agent 读了 schema_map 但没有执行任何 SQL | **否** | 只是读了，没有验证 |
| agent 执行 SQL 报错 `Unknown column` | **是，β+1**（如果用了知识中的列名） | 工具返回了明确的否定证据 |
| agent 完成任务但没用到某条知识 | **否** | 与该知识无关 |
| Gate 2 ALIGNMENT 纠正了已有知识 | **是，β+1**（被纠正的条目） | 另一条知识推翻了它 |

### 2.2 核心判定规则

**α+1 的条件**：知识被引用，且引用它的操作产生了正确结果（工具无报错、结果与预期一致）。

**β+1 的条件**：知识被引用，且引用它的操作产生了错误（工具报错、Gate 2 纠正、用户明确指出错误）。

**不计数的条件**：知识未被引用，或被引用但无法判定结果正确与否。

## 3. 按知识类型的反馈来源

不同类型的知识，其反馈来源天然不同：

### 3.1 schema（表结构）

存储内容示例：`t_room JOIN t_building ON room.building_id = building.id`

| 反馈信号 | 判定方式 | 类型 |
|---------|---------|------|
| SQL 执行成功，涉及该 JOIN 路径 | `db_query.py` 返回 exit code 0 | α+1 |
| SQL 报 `Unknown column` / `Table doesn't exist` | `db_query.py` 返回错误，且错误涉及知识中记录的列名/表名 | β+1 |
| `fetch_structure.py` 验证结构与知识一致 | 结构查询结果与知识条目匹配 | α+1 |
| `fetch_structure.py` 发现结构已变（列名改了、表被删了） | 结构查询结果与知识条目矛盾 | β+1 |

**可靠度：高**。schema 类知识的反馈信号最硬 — SQL 要么能跑通要么报错，无歧义。

### 3.2 business_rule（业务规则）

存储内容示例：`room_type 枚举值: together / single_room / double_room / ...`

| 反馈信号 | 判定方式 | 类型 |
|---------|---------|------|
| 查询中使用了该枚举值，返回了预期结果 | `WHERE room_type = 'single_room'` 有数据返回 | α+1 |
| 查询中使用了该枚举值，返回空结果或异常值 | `WHERE room_type = 'quad_room'` 返回 0 行（v2 R3 实际遇到的情况） | β+1 |
| Gate 2 ALIGNMENT 纠正了枚举值 | 新发现与已记录的枚举值矛盾 | β+1（被纠正的条目） |

**可靠度：中**。返回空结果不一定是规则错误（可能只是该值的数据为空），需要结合上下文判断。

### 3.3 tool_experience（工具经验）

存储内容示例：`t_room.rent_status 全为 NULL，不能用来判断入住`

| 反馈信号 | 判定方式 | 类型 |
|---------|---------|------|
| agent 因为知道这个坑而避免了错误查询 | agent 在推理中引用了该经验并选择了替代方案，替代方案成功 | α+1 |
| agent 忽略了该经验，踩了同样的坑 | 同样的错误再次出现 | 不计（agent 没用这条知识） |
| 该经验描述的情况已不成立（列不再是 NULL） | `fetch_structure.py --sample` 显示列有值了 | β+1 |

**可靠度：中低**。"因为知道坑而避免"这个因果链很难自动判定，可能需要 agent 自己报告。

### 3.4 query_pattern（查询模板）

存储内容示例：`统计各楼栋入住率 → SELECT b.name, COUNT(DISTINCT es.room_id)...`

| 反馈信号 | 判定方式 | 类型 |
|---------|---------|------|
| 复用该模板执行成功 | 基于模板生成的 SQL exit code 0 且结果合理 | α+1 |
| 复用该模板执行失败 | SQL 报错 | β+1 |
| 复用该模板但需要大幅修改才能跑通 | 改动量超过模板的 50% | 不计（不是直接复用） |

**可靠度：高**。SQL 模板的反馈和 schema 一样硬。

### 3.5 data_range / data_snapshot

存储内容示例：`t_employee 共 3,892 条记录（2026-03-12 统计）`

| 反馈信号 | 判定方式 | 类型 |
|---------|---------|------|
| 重新统计结果与记录一致 | `SELECT COUNT(*)` 结果在 ±5% 范围内 | α+1 |
| 重新统计结果与记录偏差大 | 差异超过 ±5% | β+1 |

**可靠度：高**。数字对比，无歧义。但这类知识 λ_base 本身就很大（0.035~0.050），衰减已经很快，贝叶斯调整的边际价值较小。

## 4. 反馈信号的可靠度分级

综合上面的分析，按可靠度排序：

```
硬反馈（自动判定，无歧义）
  ├─ SQL 执行成功/失败        → schema, query_pattern
  ├─ 结构查询比对              → schema
  └─ 数值统计比对              → data_range, data_snapshot

软反馈（需要上下文判断）
  ├─ 枚举值查询返回空结果      → business_rule
  ├─ Gate 2 ALIGNMENT 纠正    → 所有类型
  └─ 用户明确指出错误          → 所有类型

隐式反馈（因果链难以自动判定）
  └─ agent 引用了经验避免了坑  → tool_experience
```

**设计建议**：第一版只实现硬反馈，这已经覆盖了 schema、query_pattern、data_range/data_snapshot 四种类型。软反馈和隐式反馈留到后续迭代。

## 5. 元数据格式扩展

### 5.1 当前格式

```html
<!-- decay: type=schema confirmed=2026-03-12 C0=1.0 -->
```

### 5.2 扩展后格式

```html
<!-- decay: type=schema confirmed=2026-03-12 C0=1.0 alpha=0 beta=0 -->
```

新增两个字段：
- `alpha`：成功引用次数，初始值 0
- `beta`：失败引用次数，初始值 0

公式自带 +1 平滑（`(β+1)/(α+1)`），所以 alpha=0, beta=0 时 λ_effective = λ_base × 1 = λ_base，行为与当前版本完全一致。**向后兼容**：旧格式（无 alpha/beta）等价于 alpha=0, beta=0。

### 5.3 计算示例

以 schema 类型（λ_base=0.003）为例，confirmed 后 100 天：

| 使用记录 | alpha | beta | (β+1)/(α+1) | λ_eff | C(100) |
|---------|-------|------|-------------|-------|--------|
| 从未使用 | 0 | 0 | 1.0 | 0.003 | 0.741 |
| 5 次全对 | 5 | 0 | 1/6 ≈ 0.167 | 0.0005 | 0.951 |
| 3 对 2 错 | 3 | 2 | 3/4 = 0.75 | 0.00225 | 0.799 |
| 0 对 3 错 | 0 | 3 | 4.0 | 0.012 | 0.301 |

## 6. 写入时机与职责

### 6.1 谁来写入 α/β？

**Gate 4 FRESHNESS 自身**。当前 Gate 4 在读取时计算 C(t) 做三级响应，在写入时赋予 decay 标签。扩展后，Gate 4 在读取路径中增加一步：**如果本次使用产生了可判定的结果，更新 α 或 β**。

### 6.2 写入时机

```
agent 读取知识条目
    │
    ▼
agent 基于知识执行操作（SQL 查询等）
    │
    ▼
操作产生结果
    │
    ├─ 成功（exit code 0 且结果合理）→ 该知识条目 alpha+1
    ├─ 失败（工具报错且涉及该知识）  → 该知识条目 beta+1
    └─ 无法判定                       → 不更新
    │
    ▼
在当前轮次结束时（不是每次操作后），批量更新 decay 标签
```

**为什么批量而非即时？** 一个轮次中可能多次引用同一条知识，应该在轮次结束时汇总写入，避免频繁修改文件。

### 6.3 与五道门流程的集成点

```
现有 Gate 4 (read):
  C(t) = C0 × e^(-λ × t)       ← 只用 λ_base
  三级响应：TRUST / VERIFY / REVALIDATE

升级后 Gate 4 (read):
  λ_eff = λ_base × (β+1)/(α+1)  ← 新增：读取 alpha/beta 计算 λ_eff
  C(t) = C0 × e^(-λ_eff × t)    ← 用 λ_eff 替代 λ_base
  三级响应：TRUST / VERIFY / REVALIDATE（阈值不变）

新增 Gate 4 (feedback):            ← 全新路径
  轮次结束时：
  → 遍历本轮引用过的知识条目
  → 对每条知识，判定操作结果
  → 更新 alpha/beta
  → 写回 decay 标签
```

## 7. 边界情况

### 7.1 confirmed 日期是否需要更新？

**不更新**。α/β 调整的是 λ 而不是 t 的起点。confirmed 仍然记录"最后一次被工具验证为正确"的日期。只有在 REVALIDATE 触发并通过后才重置 confirmed + 清零 α/β。

### 7.2 REVALIDATE 后 α/β 是否清零？

**是**。REVALIDATE = 用工具重新验证了知识的正确性，等价于"重新确认"。此时：
- confirmed 更新为当前日期
- C0 重置为 1.0
- alpha 重置为 0
- beta 重置为 0

这确保了重新验证后知识从全新状态开始积累反馈。

### 7.3 Gate 2 ALIGNMENT 纠正时怎么处理？

被纠正的旧条目：beta+1（它被证明是错的）。
纠正后的新条目：alpha=0, beta=0（新知识，从头开始）。

### 7.4 同一轮次多次成功/失败怎么计？

每次独立的操作结果计一次。一个轮次中用同一条 schema 知识跑了 3 条 SQL 都成功，alpha+3。

### 7.5 agent "差点用错但自己纠正了"算什么？

不计。只有实际执行的操作结果才算反馈。agent 内部的推理过程不产生 α/β。

## 8. 实施路径

### Phase 1：格式扩展 + 硬反馈（最小可行）

- 扩展 decay 标签格式，加入 alpha/beta
- 只实现 SQL 执行成功/失败的自动反馈
- 覆盖：schema、query_pattern
- SKILL.md 中 Gate 4 公式升级
- 在 experiments/03 中验证

### Phase 2：软反馈

- Gate 2 ALIGNMENT 纠正时自动 beta+1
- 枚举值查询返回空结果时 beta+1（需要上下文判断逻辑）
- 覆盖：business_rule

### Phase 3：隐式反馈

- agent 自报告机制（"我引用了 X 条经验避免了 Y 错误"）
- 覆盖：tool_experience
- 这是最难的部分，可能需要 prompt 层面的设计

## 9. 开放问题

1. **α/β 是否需要上限？** 数学上不需要（(β+1)/(α+1) 天然收敛），但实际中 alpha=10000 的条目是否有意义？是否需要某种归一化或衰减窗口（只看最近 N 次）？

2. **跨会话的 α/β 累积**：当前每个会话是独立的 Claude Code 实例。α/β 写入文件后自然跨会话持久化，但如何确保并发会话不冲突？

3. **反馈的粒度**：一条 schema_map 条目可能包含多个 JOIN 路径。SQL 用了其中一个 JOIN 成功了，是整条 alpha+1 还是只给那个 JOIN 路径 alpha+1？当前 decay 标签是按条目粒度的，如果要细到字段级别，格式会变复杂。

4. **负反馈的权重是否应该更大？** 当前 α 和 β 在公式中权重对称。但现实中一次失败的信息量可能大于一次成功（知识"被证伪"比"未被证伪"信息量更大）。是否需要 `(w_β × β+1)/(α+1)` 引入不对称权重？
