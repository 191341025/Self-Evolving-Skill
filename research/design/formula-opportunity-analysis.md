# LLM 判定点公式化机会分析

> Self-Evolving Skill 设计文档
> 状态：已完成（7 个判定点全部评估，含跨点结构化分析）
> 前置依赖：decay-model-notes.md（衰减公式）、bayesian-feedback-design.md（反馈机制）

---

## 1. 立意

### 1.1 核心矛盾

SKILL.md 通过自然语言指令驱动 LLM 完成知识管理。LLM 的灵活性是核心能力，也是稳定性风险——同一条指令在不同上下文中可能产生不同判定。

### 1.2 设计思路

数学公式如同优秀的 API 函数。传统编程通过设计模式让函数相互配合，达到健壮、可维护、可扩展的效果。同理，在自然语言指令中适当引入公式，可以给 LLM 的判断加一个锚——让它在确定的参考系内发挥灵活性，而非无约束地"感觉"。

衰减公式是第一个实例：LLM 拿到 C(t)=0.73，按规则行动，比"感觉这条知识可能有点旧了"稳定得多。

### 1.3 评估原则

- **公式不替代判断，而是锚定判断。** 给 LLM 一个确定的数字或分级，收窄决策空间。
- **复杂度必须 earn its place。** 只有改善是决定性的、肉眼可见的，才值得引入。边际提升不抵复杂度代价。
- **LLM 做判断，Python 做计算。** 公式下沉到工具层，SKILL.md 只定义调用时机和响应规则。

### 1.4 与已有公式化成果的关系

衰减模型 `C(t) = C0 × e^(-λ_base × (β+1)/(α+1) × t)` 覆盖了知识生命周期中"反馈→衰减→使用"这一段。本文档评估**剩余路径节点**的公式化机会。

## 2. 知识生命周期路径

```
注入 → 存储 → 检索 → 使用 → 反馈 → 衰减 → 处置
                               ├────────┘
                               已公式化
```

公式覆盖了右半段。左半段（注入→存储→检索）和两个衔接点（反馈信号归类、VERIFY 响应）仍由 LLM 自然语言判定驱动。

## 3. LLM 判定点清单

### 3.1 Gate 1 VALUE — "这条信息值得保留吗？"

- **所在节点：** 注入
- **当前机制：** 四类高价值知识分类（结构性/行为性/异常性/演化性）+ 自然语言判定
- **LLM 具体行为：** 判断新发现的信息是否属于四类之一，不属于则拒绝存入
- **参考文档：** knowledge-taxonomy.md

### 3.2 Gate 2 ALIGNMENT — "这和已有知识矛盾吗？"

- **所在节点：** 注入
- **当前机制：** LLM 将新信息与 references/ 已有条目逐条比对
- **LLM 具体行为：** 语义比较，判定是否矛盾；若矛盾，旧条目 β+1，新条目替换

### 3.3 Gate 3 REDUNDANCY — "这条知识已经存在了吗？"

- **所在节点：** 注入
- **当前机制：** LLM 语义相似度判断
- **LLM 具体行为：** 判断新信息与已有条目是否语义重复，若重复则丢弃

### 3.4 Gate 5 PLACEMENT — "放到哪个文件/分类？"

- **所在节点：** 存储
- **当前机制：** 知识类型→文件的固定映射规则
- **LLM 具体行为：** 根据知识类型将条目写入对应 references/ 文件

### 3.5 知识检索 — "当前任务需要加载哪些知识？"

- **所在节点：** 检索
- **当前机制：** 按文件加载，LLM 根据任务上下文选择
- **LLM 具体行为：** 决定为当前调查任务加载哪些 references/ 文件；知识库小时可全量加载，增长后需要选择性加载

### 3.6 反馈信号归类 — "这次操作结果算 success 还是 failure？"

- **所在节点：** 反馈
- **当前机制：** SKILL.md 中的信号质量规则（硬信号/软信号/隐式信号分级）
- **LLM 具体行为：** 判断工具操作结果是否构成对某条知识的有效反馈，以及归类为 success 或 failure
- **参考文档：** bayesian-feedback-design.md §2-4

### 3.7 VERIFY 响应 — "C(t) 说 VERIFY，要不要真的去验证？"

- **所在节点：** 使用
- **当前机制：** 自然语言指令"VERIFY 区间先验证再使用"
- **LLM 具体行为：** 当知识置信度处于 VERIFY（0.5 ≤ C < 0.8）时，决定是花成本验证还是直接使用

## 4. 评估框架

对每个判定点，按五个维度评估：

| 维度 | 核心问题 |
|------|---------|
| **稳定性风险** | LLM 在此判定的一致性如何？同一输入是否会产生不同判定？ |
| **错判代价** | 判定错误的后果是什么？可逆还是不可逆？ |
| **公式可行性** | 是否存在可量化的输入？能否产出确定性的输出？ |
| **改善幅度** | 公式介入相比纯自然语言，改善是边际的还是决定性的？ |
| **复杂度代价** | 引入公式需要多少额外的工具/数据/计算？ |

**决策规则：** 只有稳定性风险高、错判代价大、公式可行、改善决定性四项同时满足，才值得公式化。任一项不满足，保持自然语言指令。

## 5. 评估结果

| # | 判定点 | 状态 | 结论 |
|---|--------|------|------|
| 3.1 | Gate 1 VALUE | **已评估** | 保持现状；四分类框架已是足够锚点，false positive 有衰减兜底 |
| 3.2 | Gate 2 ALIGNMENT | **已评估** | 不公式化；稳定性风险在"部分矛盾"场景，解法是结构化（实体标签+逐字段比对）而非公式 |
| 3.3 | Gate 3 REDUNDANCY | **已评估** | 不公式化；"粒度不同的重复"判定不稳定，解法同 Gate 2，靠结构化降低语义比较难度 |
| 3.4 | Gate 5 PLACEMENT | **已评估** | 保持现状；已有确定性映射规则，无需公式或结构化 |
| 3.5 | 知识检索 | **已评估** | 不需新公式；C(t) 复用为加载过滤器（TRUST > VERIFY > REVALIDATE）；相关性问题靠结构化元数据（实体标签）解决，非公式问题 |
| 3.6 | 反馈信号归类 | **已评估** | Phase 1 硬信号不加权；Phase 2 软信号引入 w_soft=0.3 加权；VPRM 已隐含于现有设计；归因置信度延后至 v3 |
| 3.7 | VERIFY 响应 | **已评估** | 保持现状，不细分 VERIFY 区间；细分本质是"用枚举代替抽象"，三级分类已是最小必要离散化 |

## 6. 跨点发现：三种锚定工具

评估 7 个判定点后，发现收窄 LLM 决策空间的手段不只是公式，而是三种工具各有适用场景：

```
公式锚定   → 用确定性数字约束判断 → 适用于可量化、有明确输入输出的环节
结构化锚定 → 用格式和元数据约束判断 → 适用于需要匹配、比对、过滤的环节
自然语言   → 不约束，依赖 LLM 灵活性 → 适用于原则性判定、语义理解
```

**核心认知：数学只适合在它应该出现的地方发挥作用，不能强加。结构化是公式之外的另一种锚定手段，两者互补而非替代。最终目的是找到每个判定点相对最适合的方式，朝最优解无限靠近。**

### 6.1 各判定点的改善方向归类

| 改善方向 | 判定点 | 具体内容 |
|---------|--------|---------|
| 公式介入 | 3.6（唯一） | Phase 2 软信号 w_soft=0.3 加权更新 |
| 现有公式复用 | 3.5 | C(t) 作为检索加载的优先级过滤器 |
| 结构化 | 3.2, 3.3, 3.5 | entities 标签实现实体级匹配 |
| 保持现状 | 3.1, 3.4, 3.7 | 现有设计已足够 |

### 6.2 结构化设计：知识条目 entities 标签

**目的：** 为 Gate 2 ALIGNMENT、Gate 3 REDUNDANCY、知识检索提供确定性的实体匹配锚点，将 LLM 的搜索空间从 O(n) 降到 O(相关子集)。

**格式：** 在 decay 标签下方新增一行 entities 标签。

```markdown
<!-- decay: type=schema confirmed=2026-03-12 C0=1.0 alpha=0 beta=0 -->
<!-- entities: t_room, t_building -->
t_room 通过 building_id 关联 t_building，但 building_id 在某些老数据中可能为 NULL
```

**粒度决策：只记实体名（表名/SP名/业务概念），不记字段名。**

理由：
- entities 标签的目的是**过滤**，不是完整描述——先缩小范围，再由 LLM 做语义比较
- 表名是数据库中最稳定的标识符，字段变化频繁会导致标签过时
- 提取表名几乎不出错，提取"哪些字段是关键的"需要判断，引入不稳定性
- 一张表相关的知识通常不超过 5-10 条，LLM 在此范围内语义比较完全胜任

**各知识类型的 entities 内容：**

| 知识类型 | entities 填什么 | 示例 |
|---------|----------------|------|
| schema | 涉及的表名 | `t_room, t_building` |
| business_rule | 表名 + 业务概念 | `t_order, refund` |
| tool_experience | 表名或工具名 | `t_room, rent_status` |
| query_pattern | 涉及的表名 | `t_room, t_building, t_employee` |

**写入时机：** inject 时 LLM 从知识内容中提取实体名，填入 entities 标签。

### 6.3 结构化设计：SKILL.md 指令风格

**原则：流程性判定用步骤式指令，原则性判定用描述式指令。**

| 判定点 | 性质 | 指令风格 |
|--------|------|---------|
| Gate 1 VALUE | 原则判定 | 描述式（四分类框架） |
| Gate 2 ALIGNMENT | 流程判定 | **步骤式**（提取entities→匹配→比对→处理） |
| Gate 3 REDUNDANCY | 流程判定 | **步骤式**（提取entities→匹配→判重→处理） |
| Gate 5 PLACEMENT | 查表操作 | 描述式（映射规则） |
| 知识检索 | 流程判定 | **步骤式**（scan→过滤→匹配→加载） |
| 反馈信号归类 | 流程判定 | **步骤式**（识别信号→分级→加权更新） |
| VERIFY 响应 | 原则判定 | 描述式（现有规则） |

**实施时机：** SKILL.md 重写随实现统一进行，不提前改写。原因：entities 标签和 w_soft 加权尚未实现，步骤式指令需要引用这些基础设施。

### 6.4 设计反模式警示

**"用枚举代替抽象"反模式：** 在评估 3.7 VERIFY 响应时确认——当一个连续模型（C(t)）已经提供了统一的量化框架时，继续在其上叠加离散分类（细分 VERIFY 为 VERIFY-HIGH / VERIFY-LOW）是危险的设计方向。三级分类（TRUST/VERIFY/REVALIDATE）是最小必要离散化，进一步细分会导致分支爆炸和维护成本指数增长。
