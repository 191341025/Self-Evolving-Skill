## 实验过程中的错误

| 轮次 | 错误 | 根因 | 解决方式 |
|------|------|------|---------|
| R1 | `Unknown column 'c.referenced_table_name'`（information_schema） | 查询对当前MySQL变体过于复杂 | 简化查询 |
| R2 | `Unknown column 'b.type'` | 假设了列名 | 发现实际列名为 `building_type` |
| R2 | `Unknown column 'record_time'` | 假设了列名 | 发现实际列名为 `auth_time` |
| R4 | 中文字符串在 WHERE 中导致 exit code 127 | Shell 编码踩坑 | 改用数值字段（已知踩坑经验，第二次尝试时避开） |

所有错误均为瞬时性（在同一任务内解决），没有导致错误知识被持久化。

## 总体总结

### 审计结论：通过

全部六个审计维度均健康：

| 维度 | 评分 | 说明 |
|------|------|------|
| 拒绝率 | ✅ | 76%拒绝率（19拒绝/25评估）— 严格过滤 |
| 准确性 | ✅ | 1次自我修正（room_type），最终状态0错误 |
| 冗余度 | ✅ | 0条冗余；Gate 3 无需合并 |
| 膨胀度 | ✅ | 176行/5文件；最大文件54行，远在限制内 |
| 衰减标注 | ✅ | 所有条目携带完整衰减元数据；置信度计算已验证 |
| 放置准确 | ✅ | 结构→schema_map，规则→business_rules，踩坑→query_patterns |

### 核心发现

1. **76%拒绝率证实严格治理**：仅约24%的候选知识通过全部五道门，确保知识库保持精简和准确。

2. **Gate 2 ALIGNMENT 自我修正跨实验有效**：v1 修正了 status 规则；v2 修正了 room_type 枚举。该门控持续捕获并修复不准确条目。

3. **置信度衰减模型增加价值而不增加膨胀**：总体量（176行）与v1（约170行）相当，但每条知识现在携带元数据，支持未来的时间感知决策。

4. **R5 重复任务轮验证知识复用**：4/4任务首次尝试成功，零 fetch_structure 调用，证明持久化知识充分且正确。

5. **investigation_flows 保持仅模板状态**：与v1不同（v1添加了实际流程），v2的任务组合未产生可复用的多步骤模式。这是可接受的 — 模板已为未来使用做好准备。

6. **从v1继承的知识得到验证**：3条 tool_experience 条目继承自v1，维持 TRUST 置信度（C=0.98，4天后），被正确复用而无需重新确认。
