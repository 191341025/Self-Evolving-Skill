# Knowledge Decay Governance — Implementation Spec
> 本文档是写给 Claude Code 的工程规格说明。
> 目标：将知识置信度衰减模型融合进 Self-Evolving Skill 的 Gate 4（FRESHNESS）治理层，
> 实现从"被动日期标注"到"主动置信度管理"的升级。

---

## 背景与动机

当前 Gate 4 的工作方式：
- 时间敏感的知识写入时标注日期（如 `2026-03-08 确认`）
- 依赖后续会话"碰巧"发现过期数据后手动清理
- 问题：知识老化是一个**连续过程**，但现在只有"有效"和"被发现过期"两个状态

升级目标：
- 每条知识携带一个**置信度** `C ∈ [0, 1]`
- 置信度随时间自动衰减，不同类型的知识衰减速率不同
- 当置信度低于阈值时，触发对应级别的验证行为
- Gate 4 从"打标签"升级为"管置信度"

---

## 核心公式

```
C(t) = C₀ × e^(-λ × t)

C(t)  = t 天后该条知识的置信度，范围 [0, 1]
C₀    = 写入时的初始置信度，通常为 1.0
λ     = 衰减速率，由知识类型决定（见下表）
t     = 距上次确认的天数

半衰期 t½ = ln(2) / λ ≈ 0.693 / λ
```

---

## λ 值参考表

| 知识类型 | λ 值 | 半衰期 | 说明 |
|---------|------|--------|------|
| 表结构 / JOIN 路径 | 0.003 | ~231天 | 结构变化极少，高度稳定 |
| 业务规则 / 枚举值 | 0.008 | ~87天 | 偶有修正，较稳定 |
| 工具使用经验 | 0.005 | ~139天 | 跨库可迁移，衰减慢 |
| 查询模板 / SQL | 0.015 | ~46天 | 依赖表结构，结构变则模板失效 |
| 数据时间范围 | 0.035 | ~20天 | 数据持续增长，需定期更新 |
| 数量统计 / 分布 | 0.050 | ~14天 | 随业务变化快，几乎不入库 |

> **注意**：λ 值为当前经验估算值，待 Experiment #02（电信计费库）数据后校准。
> 如果两个领域的拒绝模式产生接近的 λ 分布，说明该表具有跨域普适性。

## 知识条目的新格式

在 `references/` 的任意主题文件中，每条知识从现在起采用以下格式：

```markdown
<!-- decay: type=<知识类型> confirmed=<YYYY-MM-DD> C0=1.0 -->
<知识内容>
```

**示例（schema_map.md）：**
```markdown
<!-- decay: type=schema confirmed=2026-03-08 C0=1.0 -->
employee → record: t_emp_record.id_card = t_employee.id_card

<!-- decay: type=data_range confirmed=2026-03-08 C0=1.0 -->
t_emp_record 数据范围：2025-12-10 ~ 2026-02-28（可能持续增长）
```

**type 字段的合法值：**
- `schema` → λ=0.003
- `business_rule` → λ=0.008
- `tool_experience` → λ=0.005
- `query_pattern` → λ=0.015
- `data_range` → λ=0.035
- `data_snapshot` → λ=0.050

## 三级触发机制

Claude Code 在**读取** references/ 中任意知识条目时，先计算当前置信度：

```
t     = today - confirmed（天数）
C(t)  = C0 × e^(-λ × t)
```

然后根据置信度等级决定行为：

### Level 1 — TRUST（C ≥ 0.8）
- 直接使用，无需任何额外操作
- 不在响应中提及置信度

### Level 2 — VERIFY（0.5 ≤ C < 0.8）
- 使用该知识，但在响应末尾追加一条提示：
  ```
  ⚠️ [知识类型] 距上次确认已 N 天，建议本次会话顺手验证是否仍然准确。
  ```
- 如果会话中有机会顺手查询验证，主动去验证
- 验证通过后，更新 `confirmed` 日期，置信度重置为 1.0

### Level 3 — REVALIDATE（C < 0.5）
- **不直接使用该知识**，先执行验证
- 使用工具（如 db_query.py / fetch_structure.py）查询最新数据
- 验证结果与已有知识比对：
  - 一致 → 更新 `confirmed`，置信度重置为 1.0，然后使用
  - 矛盾 → 触发 Gate 2（ALIGNMENT），修正知识条目，然后使用
  - 无法验证 → 告知用户该知识可能已过期，请求确认

## 对 _index.md 的修改

在路由表中新增一列，显示每个文件的最低置信度（即文件内所有知识中置信度最低的一条）：

```markdown
| Topic File             | Summary        | When to Load     | Min Confidence |
|------------------------|----------------|------------------|----------------|
| schema_map.md          | 表结构与关联关系 | 需要知道数据在哪里  | 0.94           |
| query_patterns.md      | 可复用SQL模板   | 执行数据查询时     | 0.82           |
| business_rules.md      | 业务规则与枚举  | 分析业务逻辑时     | 0.91           |
| investigation_flows.md | 多步调查流程    | 复杂调查时        | 0.88           |
```

`Min Confidence` 由 Claude Code 在每次更新知识文件后自动重新计算并写入。
这样在路由阶段就能感知到哪些文件需要重点关注。

## 对 Gate 4 的修改（SKILL.md 更新）

将 SKILL.md 中的 Gate 4 描述从：

```
Gate 4 — FRESHNESS: 时间敏感数据？
  → 是 → 标注日期 (YYYY-MM-DD)
  → 否 → 稳定事实入库
```

升级为：

```
Gate 4 — FRESHNESS: 写入时
  → 判断知识类型（schema / business_rule / tool_experience /
                  query_pattern / data_range / data_snapshot）
  → 写入格式：<!-- decay: type=<type> confirmed=<YYYY-MM-DD> C0=1.0 -->
  → 高衰减类型（data_range / data_snapshot，λ≥0.035）
     优先考虑拒绝（Gate 1 已拦截的通常不会到这里）

Gate 4 — FRESHNESS: 读取时
  → 计算 C(t) = e^(-λ×t)
  → C≥0.8  → TRUST，直接使用
  → 0.5≤C<0.8 → VERIFY，使用并提示
  → C<0.5  → REVALIDATE，先验证再使用
```

- 与 λ 值对照表比对，校准不准确的 λ 值

## 边界约定

- **不自动删除**任何知识条目，即使 C(t) → 0。删除需要用户确认。
- **λ 值不可在会话中修改**，修改 λ 值需要人工更新本文档后重新实施。
- **置信度重置条件**：只有通过工具实际查询验证后才能重置，不能仅凭"感觉一致"重置。
- **Gate 2 优先于衰减**：如果 REVALIDATE 阶段发现矛盾，走 Gate 2 流程，不走衰减修复流程。

## 预期效果

| 指标 | 当前行为 | 升级后行为 |
|------|---------|-----------|
| 过期知识发现 | 被动，依赖碰巧 | 主动，置信度触发 |
| Gate 4 精度 | 二元（有效/无效） | 连续（0~1 置信度） |
| 错误知识存活时间 | 不确定 | 受 λ 控制，可预测 |
| 跨库知识迁移 | 手动判断 | λ 分类自动指导 |
| 实验可比性 | 仅看拒绝率 | 可追踪置信度分布 |
